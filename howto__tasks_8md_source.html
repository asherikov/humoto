7 Types of tasks {#md_howto_tasks_types}
8 ==============
9 
10 Tasks are composed of two main elements
11 
12 > * body -- a matrix, which multiplies the vector of decision variables 'x';
13 > * bounds -- a vector(s) which constrain result of multiplication of the body
14 > by 'x'.
15 
16 
17 Currently, there are four types of task bodies:
18 
19 > * **A*x** -- 'x' is multiplied by a dense matrix 'A';
20 > * **A*S*x** -- a continuous segment of 'x' is selected by matrix S and then
21 > mupltiplied by a dense matrix 'A';
22 > * **G*I*x** -- a set of individual values of 'x' is selected by matrix 'I' and
23 > then each of the values is multiplied by a corresponding scalar gain stored
24 > in matrix 'G';
25 > * **I*x** -- 'I' selects values of 'x', no weighting is performed, such tasks
26 > are assumed to be satisifed exactly.
27 
28 Note that the structured matrices 'S', 'G', 'I' are not stored or defined
29 explicitly. Refer to the documentation of tasks to learn how to specifiy them.
30 
31 
32 Bounds can also be of different types:
33 > * (L<=, <=U) -- lower 'L' and upper 'U' bounds are defined;
34 > * (<=U) -- only upper 'U' bound is defined;
35 > * (L<=) -- only lower 'L' bound is defined;
36 > * (=B) -- equality task: 'B = L = U';
37 > * (=0) -- equality task with 'B = 0';
38 
39 
40 Various combinations of task bodies and bounds produce the following
41 [task types](@ref BaseTasks):
42 
43 | | Equality (=0) | Equality (=B) | Lower bounds (L<=) | Upper bounds (<=U) | Lower and upper bounds <br/> (L<=, <=U)
44 |-----------|-----------------------|----------------------|----------------------|----------------------|----------------------------------
45 | **A*x** | @ref humoto::TaskAB0 | @ref humoto::TaskAB | @ref humoto::TaskAL | @ref humoto::TaskAU | @ref humoto::TaskALU
46 | **A*S*x** | @ref humoto::TaskASB0 | @ref humoto::TaskASB | @ref humoto::TaskASL | @ref humoto::TaskASU | @ref humoto::TaskASLU
47 | **G*I*x** | @ref humoto::TaskGIB0 | @ref humoto::TaskGIB | @ref humoto::TaskGIL | @ref humoto::TaskGIU | @ref humoto::TaskGILU
48 | **I*x** | @ref humoto::TaskIB0 | @ref humoto::TaskIB | @ref humoto::TaskIL | @ref humoto::TaskIU | @ref humoto::TaskILU

52 Implementation {#md_howto_tasks_impl}
53 ==============
54 
55 Each user-defined task must inherit from one of the task classes whose names
56 are given in the table in the previous section. It is also possible to derive
57 from one of [predefined tasks](@ref PredefinedTasks) instead.
58 
59 Each user-defined task must have a unique string id (within a hierarchy) and
60 implement method @ref humoto::TaskBase::form(), which should
61 
62 * initialize body and bounds of the task;
63 * multiply generated body and bounds by the task gain.

67 Performance {#md_howto_tasks_perf}
68 ===========
69 
70 Choosing task type {#md_howto_tasks_perf_type}
71 ------------------
72 
73 In order to attain the highest performance it is necessary to carefully choose
74 a parent class. While choosing a parent class the programmer should follow
75 several principles:
76 
77 * Pick the most specific parent class -- in the table above the most specific
78  classes are on the left and at the bottom.
79 
80 * Do not perform any task transformations manually. In particular, do not
81  convert upper bounds to lower bounds and vice versa, do not express two-sided
82  inequalities as constraints with only lower or upper bounds. All
83  transformations are performed automatically when necessary.
84 
85 * If a task may change its form from one control iteration to another, pick the
86  most specific parent class that represents all forms or implement multiple
87  tasks.

90 Active set guessing {#md_howto_task_perf_guess}
91 -------------------
92 
93 For the sake of performance, inequality tasks may also implement
94 @ref humoto::TaskBase::guessActiveSet() method, which should generate guess of
95 active constraints. A default implementation of this method is provided for all
96 tasks and uses the following heuristics
97 
98 * if the actual active set from the previous iteration is empty, mark all
99  constraints as inactive;
100 
101 * if the actual active set from the previous iteration is not empty, copy the
102  actual active set from the previous iteration into guessed active set using
103  methods @ref humoto::TaskBase::getActiveSetGuess() and
104  @ref humoto::TaskBase::getActualActiveSet().
105 
106 Alternatively, if you are working on an MPC problem, it is reasonable to shift
107 the active set from the previous iteration forward by the number of constraints
108 of the respective task in one interval, marking constraints of the last
109 interval as inactive.
