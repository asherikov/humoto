7 Compatibility {#md_coding_compat}
8 =============
9 
10 * **C++ standard**: In order to avoid problems with portability we enforce
11  C++98 standard with appropriate compilation flags. Note for developers: It
12  appears that `clang++` is sloppier than `g++` and silently accepts code which
13  results in a failure of the latter. Hence, it is recommended to verify the
14  code with `g++` before pushing changes to the master branch.
15 
16 * **cmake**: All cmake files and scripts should be compatible with version
17  2.8.12 of `cmake`.

21 General policies {#md_coding_general}
22 ================
23 
24 * Sometimes we use numbers in the names of various objects: modules, tests,
25  etc. It is desirable to make these names unique not only currently,
26  but also in the repository history. The simplest way to achieve this is to
27  assign strictly incremental numbers. For example, if there are two tests
28  `test_005` and `test_010` in a certain module, a newly added test must have
29  name `test_011` and not `test_000` nor `test_006`. There still might be a
30  collision with a deleted `test_011` -- such cases should also be avoided if
31  possible.

41 General rules {#md_coding_style_general}
42 -------------
43 
44 ### Tabulation
45 
46 > * No tabs should be used. Indentation must include 4 spaces.
47 
48 
49 ### Naming
50 
51 > * Function names, variable names, and filenames should be descriptive; avoid
52 > abbreviation. Types and variables should be nouns, while functions should
53 > be "command" verbs.
54 
55 
56 
57 Specific rules {#md_coding_style_specific}
58 --------------
59 
60 ### Files
61 
62 > * Filenames should be all lowercase and may include underscores `_`.
63 >
64 > * Extensions of C++ source and header files should be `.cpp` and `.h`
65 > respectively.
66 
67 
68 
69 ### Type names
70 
71 > Names of all types -- classes, structs, typedefs, and enums -- have the same
72 > naming convention:
73 >
74 > * Type names start with a capital letter and have a capital letter for each new
75 > word, with no underscores: `MyExcitingClass`, `MyExcitingEnum`.
76 
77 
78 
79 ### Enumerations
80 
81 > * All enumerations must be defined within some container class scope.
82 > * Names of the enumerators should be in upper case with underscores.
83 
84 
85 
86 ### Variables
87 
88 > * Variable names are all lowercase, with underscores between words, i.e.,
89 > `my_table_name`.
90 >
91 > * Global variables must be avoided. If it is not possible, their names must
92 > have `g_` prefix, e.g., `g_my_global_variable`.
93 
94 
95 
96 ### Functions and methods
97 
98 > * Functions should start with a lowercase letter and have a capital letter for
99 > each new word without underscores.
100 >
101 > * Function names should typically be imperative, i.e., they should be commands:
102 > `openFile()`.
103 >
104 > * Parameters, which do not serve as outputs, must be marked with `const`.
105 >
106 > * Output parameters must be gathered in the beginning of the list of
107 > parameters:
108 > `doSomething(output1, output2, input1, input2 = <default_value>)`.
109 
110 
111 
112 ### Classes
113 
114 > * It is not allowed to mix definitions with different access modifiers, i.e.,
115 > methods and members with the same access modifier should be gathered
116 > together.
117 >
118 > * It is recommended to use access modifiers multiple times to visually separate
119 > members from methods.
120 >
121 > * Names of member variables should have trailing underscores, i.e.,
122 > `member_name_`.
123 >
124 > * Methods, which do not change the corresponding class, must always be marked
125 > with `const`.
126 >
127 > * In general, destructors of base classes must be defined and must be
128 > protected. If it is necessary to allow polymorphic destruction, the
129 > destructor should be defined as public and virtual.
130 >
131 > * Constructors of base classes should be protected as well.
132 
133 
134 
135 ### Macro
136 
137 > * Macro should be avoided when possible.
138 > * Macro name must be in all capitals with underscores.
139 
140 
141 
142 ### Namespaces
143 
144 > * Names of namespaces should be in lower case with possible underscores.
145 
146 
147 
148 ### Templates
149 
150 > * Names of template parameters should follow the same conventions as the
151 > category they belong to (typenames, variables), but their names must include
152 > `t_` prefix: `t_BaseClass`.

156 Visibility of symbols {#md_coding_style_visibility}
157 ---------------------
158 
159 * We assume that implementation details should not be exposed when it is
160  compiled into a library. Since it is default on many platforms, we introduced
161  special macro: `HUMOTO_LOCAL`, which hides the corresponding symbol using
162  appropriate compiler attributes. The opposite effect is achieved with
163  `HUMOTO_API`.

167 Implementing tests {#md_coding_tests}
168 ==================
169 * It is preferrable to add a new regression test case to an existing related
170  regression test instead of creating a new test and thus forcing creation of
171  an additional binary.

175 Tips {#md_coding_tips}
176 ====
177 
178 ## Eigen
179 
180 * In some cases, Eigen automatically creates temporary variables in order to
181  avoid problems when the assigned variable participates in the expression on
182  the right hand side of the assignment. This behavior can be suppressed with
183  `noalias()`: `a.noalias() = b * c`. Note, however, that expressions like
184  `a.noalias() = b * a` are potentially dangerous.
185  See https://eigen.tuxfamily.org/dox/group__TopicAliasing.html for more
186  information.
